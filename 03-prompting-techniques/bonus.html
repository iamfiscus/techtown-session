Module 1 — When the build breaks

Lovable did something weird. Now what?

Everyone hits this. The screen's blank, the button does nothing, it deleted the thing you liked.

You don't need to know how to code. You need to know how to diagnose.

prompting-techniques.vercel.app/bonus2 / 36

Don't rage-prompt. Diagnose.

When something breaks, the instinct is to type "it's broken, fix it" over and over.

  • The AI isn't broken — it's missing information
  • Same lesson as V1: vague in → vague out
  • "Fix it" is a wish. Describe what you see — that's a clue.

A good bug report is just Context + Task, aimed at a problem.

prompting-techniques.vercel.app/bonus3 / 36

Describe a bug in 3 parts

  • Expected — what you thought would happen
  • Actual — what happened instead
  • On screen — what you literally see (error text, blank area, wrong color)

If there's an error message, paste the whole thing. Don't summarize it.

Specifics are clues. "It's broken" is not.

Still stuck? Flip it — the interview

When you don't even know what to ask for, make the AI ask you.

  • You stop guessing at prompt wording
  • The AI pulls the context out of you, one question at a time
  • You just answer — answering is easier than prompting

This is the 5th move from the core deck — here's the full template.

prompting-techniques.vercel.app/bonus6 / 36

Copy-paste — the interview

I want to build [PROJECT]. Before we write anything,
interview me about my project. Ask one question at a
time. Wait for my answer before the next question.
When you have enough to write a clear spec, summarize
it back to me and we'll start.

Answer honestly. The summary it gives back is your new prompt.

When to start a fresh chat

Remember the context window — long chats lose the thread.

Signs the chat has gone bad:

  • It re-introduces bugs you already fixed
  • It forgets decisions you made earlier
  • Answers get slower and vaguer

Fix: new chat. Paste your V4 prompt + a 3-line summary of where you are.

Module 2 — After V4 works

It works. Don't break it.

You've got something real on screen. Now you're scared to touch it.

Iterating is a skill. It's not "ask for everything at once."

prompting-techniques.vercel.app/bonus10 / 36

Small, reversible changes

  • One thing at a time — so you know what caused what
  • Reversible — you can always say "undo that last change"
  • Name what's working — tell it what NOT to touch

Big-bang changes are how working apps get broken.

prompting-techniques.vercel.app/bonus11 / 36

Three ways to move forward

  • Change one section — "only change the booking form, leave the rest"
  • Layer one feature — add features one prompt at a time, not a list of ten
  • Ask for variations — "give me 3 versions of this screen, then I'll pick"

Each prompt = one move.

prompting-techniques.vercel.app/bonus12 / 36

Copy-paste — change one thing

# Change
[the one thing to change]

# Leave alone
Everything else is working — don't touch it.

# Check
After the change, tell me what you modified
and confirm nothing else moved.

Changing how it looks

Design feedback works when you describe a vibe, not a hex code.

Vague

"Make it look better"

"Nicer colors"

A vibe it can use

"More premium and calm — like a high-end spa"

"More trustworthy, less playful"

Name a feeling and a reference. It'll translate that to design.

Module 3 — Data & backends

The wall everyone hits.

Your app looks great — then you refresh the page and everything you typed is gone.

You need a backend. You don't need to understand one.

prompting-techniques.vercel.app/bonus17 / 36

What a backend actually is

In plain terms:

A place your app saves stuff so it's still there tomorrow.

  • The screen = what people see (the front)
  • The backend = where the info lives (the back)
  • No backend = nothing is remembered between visits

You don't configure it. You ask for it.

prompting-techniques.vercel.app/bonus18 / 36

Meet Supabase

Lovable has a built-in backend called Supabase.

  • When Lovable says "Supabase" — that's your database talking
  • You won't open it or set it up by hand
  • You just say what you want saved, and Lovable wires it

Recognize the word. That's all you need today.

prompting-techniques.vercel.app/bonus19 / 36

How to ask for it

  • Saving data — "add a database so appointments are saved"
  • Accounts — "users can create an account and log in"
  • Forms that stick — "when someone submits this form, save it and show it on the dashboard"

Describe the outcome in business terms. Lovable picks the tech.

Copy-paste — backend request

# What to save
[the thing — e.g. appointments, customers, orders]

# Who can see it
[just me / logged-in users / the public]

# Where it shows up
[which screen displays this saved data]

Set up the database for this and connect it to
the form and the screen. Use Supabase.

Worked example — the salon app

# What to save
Appointments — name, service, date, time

# Who can see it
Just me, the stylist — after I log in

# Where it shows up
A "Today's appointments" list on my dashboard

Set up the database for this and connect it to
the booking form and the dashboard. Use Supabase.

Now the booking is still there tomorrow morning.
